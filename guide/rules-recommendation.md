# GitHub Actions 관련 Cursor Rules 추천

## 🤔 Rules가 필요한가?

### 현재 상황 분석

**Rules를 만들 필요가 있는 경우**:
- ✅ 팀 전체가 일관된 워크플로우 작성 스타일을 유지해야 할 때
- ✅ 반복적으로 사용하는 패턴이 있을 때
- ✅ AI가 자동으로 특정 형식을 따라야 할 때

**Rules 없이도 충분한 경우**:
- ✅ 개인 프로젝트이거나 학습 단계일 때
- ✅ 유연하게 실험하고 싶을 때
- ✅ 가이드 문서만으로 충분할 때

## 💡 추천 방안

### 옵션 1: 가이드 문서 중심 (추천)

**장점**:
- 유연성: 필요할 때 참고
- 학습 친화적: 단계별 학습 가능
- 변경 용이: 문서만 수정하면 됨

**단점**:
- AI가 자동으로 적용하지 않음
- 매번 참고해야 함

**적용 시점**: 현재 단계 (학습 및 실험 단계)

### 옵션 2: Rules 추가

**장점**:
- AI가 자동으로 패턴 적용
- 일관성 보장
- 반복 작업 자동화

**단점**:
- 초기 설정 복잡
- 유연성 감소
- 학습 단계에서는 제약이 될 수 있음

**적용 시점**: 패턴이 확정된 후 (운영 단계)

## 📝 Rules 예시 (필요시 사용)

만약 Rules를 만들고 싶다면, 다음과 같은 내용을 포함할 수 있습니다:

```markdown
# GitHub Actions Rules

## 워크플로우 작성 규칙

1. 모든 워크플로우는 실행 후 구조화된 리포트를 생성해야 함
2. 리포트는 `.github/reports/` 디렉터리에 JSON 형식으로 저장
3. 모든 워크플로우는 메타데이터(실행 시간, 상태 등)를 포함해야 함
4. 실패 시 상세한 에러 정보를 로그에 출력해야 함

## 네이밍 규칙

- 워크플로우 파일: `kebab-case.yml`
- 작업 이름: `Pascal Case`
- 스텝 이름: `문장 형식` (예: "Run tests")

## 리포트 형식

워크플로우 리포트는 다음 구조를 따라야 함:
- metadata: 실행 메타데이터
- status: 전체 상태
- jobs: 작업별 상세 정보
- metrics: 성능 메트릭
```

## 🎯 최종 추천

### 현재 단계: 가이드 문서 중심

**이유**:
1. 학습 단계에서는 유연성이 중요
2. 다양한 패턴을 실험해봐야 함
3. 패턴이 확정되면 그때 Rules 추가

### 향후 단계: Rules 추가 고려

**시점**:
- 워크플로우 패턴이 3개 이상 확정되었을 때
- 팀원이 늘어나 일관성이 필요할 때
- 운영 단계로 전환할 때

## 📋 체크리스트

현재는 다음 문서들로 충분합니다:
- ✅ `guide/github-actions-guide.md` - 전체 가이드
- ✅ `guide/github-actions-implementation-plan.md` - 구현 계획
- ✅ `guide/ai-analysis-guide.md` - AI 분석 가이드

Rules는 다음 상황에서 추가 고려:
- [ ] 워크플로우 패턴이 5개 이상 확정됨
- [ ] 팀원이 2명 이상
- [ ] 운영 단계로 전환
- [ ] 일관성 문제 발생

## 💬 결론

**현재는 Rules 없이 가이드 문서만으로 진행하는 것을 추천합니다.**

이유:
- 학습 단계에서는 유연성이 중요
- 다양한 패턴 실험 필요
- 패턴 확정 후 Rules 추가가 더 효과적

**Rules 추가는 운영 단계로 전환할 때 고려하세요.**
